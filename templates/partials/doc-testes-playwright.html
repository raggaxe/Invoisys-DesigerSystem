Testes · Playwright

Os exemplos desta documentação expõem data-testid com o prefixo ds- para que equipas de QA e automação possam asserições estáveis sem depender só de texto visível ou de classes puramente visuais.

Porque testar assim

Texto de botões muda com copy e traduções; classes de tema mudam com o design system. Um data-testid dedicado por ação ou por nó crítico mantém os testes legíveis (“clicar em ds-nav-inputs”) e menos frágeis quando o CSS evolui. No portal MVC, replique o mesmo critério com prefixo por módulo (ex.: data-testid="nfe-listagem-filtro-abrir").

Boas práticas de data-testid

  • Um valor por função — um id estável por botão, campo ou região que o teste precisa de endereçar; evite duplicar o mesmo data-testid na mesma página.
  • Prefixo consistente — nesta doc, ds- (ex.: ds-nav-botoes, ds-copy-pre-intro-mvc); no produto, escolha um prefixo por área (ex.: nfe-, po-).
  • Não usar só classes utilitárias — .btn-primary pode aparecer dezenas de vezes; o teste deve mirar o nó certo.
  • Alinhar com QA — documente numa checklist ou no gestor de casos de teste quais data-testid cobrem cada cenário; o Playwright torna-se documentação executável.

Ferramentas e fluxo

Instale o Playwright no repositório de testes (npm init playwright@latest ou equivalente na vossa stack). Corra em headed durante o desenvolvimento da spec e em CI headless no pipeline; use test.step para relatórios legíveis para QA.

Exemplo real — navegar nesta documentação

Servidor local na porta por defeito do app.py (8080). Ajuste baseURL e portas ao vosso ambiente.

Playwright · TypeScript 
import { test, expect } from "@playwright/test";

test.describe("Design system doc", () => {
  test.use({ baseURL: "http://127.0.0.1:8080" });

  test("abre Botões a partir do menu", async ({ page }) => {
    await page.goto("/#introducao");
    await page.getByTestId("ds-nav-botoes").click();
    await expect(page.getByTestId("ds-panel-botoes")).toBeVisible();
    await expect(page.locator("#sec-botoes-titulo")).toContainText("Botões");
  });

  test("copiar snippet de layout MVC", async ({ page }) => {
    await page.goto("/#quick-start");
    await page.getByTestId("ds-copy-pre-intro-mvc").click();
    // Validar feedback "Copiado!" ou conteúdo no clipboard conforme o vosso critério de aceitação
  });
});

Exemplo real — view MVC com data-testid

No Razor, exponha o mesmo contrato que QA valida manualmente. O teste abre a listagem, filtra e asserí o estado da grelha sem depender da cor da linha.

Playwright · fluxo ilustrativo 
import { test, expect } from "@playwright/test";

test("listagem NF-e — filtro visível", async ({ page }) => {
  await page.goto("/NfeEntrada"); // URL ilustrativa
  await expect(page.getByTestId("nfe-listagem-filtro-form")).toBeVisible();
  await page.getByTestId("nfe-listagem-filtro-abrir").click();
  await page.getByTestId("nfe-listagem-filtro-status").selectOption("Autorizada");
  await page.getByTestId("nfe-listagem-filtro-aplicar").click();
  await expect(page.getByTestId("nfe-listagem-grid")).toBeVisible();
});

Deixar pronto para QA

  • Listar no ticket ou no caso de teste os data-testid que cobrem cada passo (dado / quando / então).
  • Garantir que builds de staging expõem os mesmos ids que o ambiente onde o Playwright corre.
  • Anexar ao relatório de sprint um link para o ficheiro de spec ou para o job de CI que executa a suite.
  • Preferir um projeto de testes partilhado (ou pacote npm) a specs soltas na máquina de um developer — reprodutibilidade para QA.
