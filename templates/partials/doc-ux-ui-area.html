UX, UI e designer — visão única

Três papéis que caminham juntos e não são intercambiáveis. Use esta secção como referência em reuniões: um slide por coluna ou a página inteira projetada.

UX

Experiência do usuário

  • Problema, jornada e tarefa — fluxos e arquitetura de informação; acessibilidade como requisito
  • Menos fricção no trabalho real (faturação, aprovações, listagens) e critérios quando der para ir além do «achismo»
  • Decisões escritas para engenharia: modais, filtros, vazios e erros definidos antes do código
  • Pergunta-guia: «Isso funciona e faz sentido para quem usa o portal?»
UI

Interface do usuário

  • Visual: tipografia, cores, ícones, espaçamento e componentes com estados (hover, erro, vazio, loading)
  • Protótipos e telas no Figma alinhados ao CSS do design system — repasse claro ao desenvolvimento
  • Previsibilidade entre módulos; evitar CSS solto na view que não escala
  • Pergunta-guia: «Isso está claro, consistente com a marca e fácil de escanear?»
Designer

Identidade e sistema visual

  • Marca, bibliotecas e tokens coerentes entre áreas (ex.: NF-e, compras, tesouraria)
  • Costura UX + UI: do wireframe à alta fidelidade, com qualidade de entrega
  • Padrões reutilizáveis — o que não entra no design system não vira exceção permanente
  • Pergunta-guia: «Isso está alinhado ao design system e à identidade Invoisys?»

Em uma frase cada: UX — caminho e comportamento; UI — aparência e toque; Designer — marca e sistema que escalam. Os três precisam estar alinhados: UI bonita sem UX esconde fricção; UX boa sem UI clara cansa; sem direção de design, o portal perde consistência entre equipes e telas.

Resumo da apresentação

  • Uma ponte, não um muro — produto, design e engenharia no mesmo desenho; menos retrabalho, entregas mais rápidas.
  • Componentes prontos — definição, CSS oficial e snippet copiável; evita refazer o mesmo botão ou modal em cada tela.
  • Linguagem visual única — padrões repetidos no portal (ex.: NF-e à tesouraria); mais confiança e menos esforço mental.
  • Documentação viva — o que vê aqui segue o MVC; regras de uso em Manifesto de UX e cultura da equipe.

Como nasce uma tela hoje

Protótipo → Figma → MVC. Seis passos; no dia a dia, as fases podem se cruzar.

1 · Produto

Feature com interações, ações e regras.

2 · Protótipo

Tela montada com componentes que já existem.

3 · Falta peça?

Novo componente: visual + CSS canônico + doc. Sistema.

4 · Figma

Biblioteca da tela + estados no Figma; repasse aos desenvolvedores.

5 · Front

View MVC: classes do design system + backend.

6 · No ar

Tela no portal; testes se aplicável — Playwright.

Design system no código: Introdução · Manifesto de UX e cultura da equipe

Contexto da equipe

O que é a área

Em linguagem simples, front-end é o que aparece no navegador quando alguém usa o portal Invoisys: telas, botões, formulários, listas e mensagens — a parte que dá para ver e clicar, não a “máquina” dos dados por trás. A nossa frente junta produto, design e engenharia para que isso fique fácil de entender, previsível de um módulo ao outro e alinhado ao trabalho real de quem usa o sistema no dia a dia.

Componentes do sistema: criação, documentação e disponibilidade

Frente de componentes compartilhados (não o layout pontual de uma tela): UI + CSS oficial + doc com snippet — só assim o padrão escala.

Criação e evolução

Acordo com produto e revisão técnica; merge no CSS compartilhado do portal, não só neste storybook.

Disponível para desenvolvedores

Classes e exemplos MVC — menos «como você fez o botão X?».

Atualização em bloco

Token ou componente atualizado beneficia todas as views alinhadas ao padrão.

Comunicar mudanças

Release notes, canais (Teams/Slack) e data-testid com QA — ver Testes · Playwright.

Ferramentas usadas

  • Figma — ferramenta que usamos para UI: bibliotecas de componentes, protótipos e entrega ao desenvolvimento (especificação para engenharia).
  • Cursor (IDE com IA) — editor no dia a dia do código e da documentação. Com o uso consistente de IA assistida, a equipe cresceu em capacidade: entregas mais rápidas e mais sólidas, revisões mais claras e documentação mais adequada ao que vai para produção.
  • Jira — backlog, épicos, tarefas e acompanhamento de entregas com produto e engenharia.
  • Confluence — wiki e documentação colaborativa (requisitos, decisões, processos).
  • Miro — quadros para fluxos, discovery, workshops e alinhamento visual entre equipes.
  • Visual Studio / Git — código e revisão do stack CSS no portal.
  • Este storybook — pré-visualização de componentes e snippets copiáveis alinhados ao MVC.
  • Playwright — regressão, fluxos críticos e data-testid (ver Testes · Playwright).
